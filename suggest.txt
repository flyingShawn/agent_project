二、当前有哪些提示词，什么时候生效，效果是什么

Agent 总控系统提示词
在每次聊天进入图时注入一次，约束整个工具选择和回答规则。prompts.py (line 25) nodes.py (line 57)
生效时机：每次 chat 请求初始化。
效果：决定优先用哪个工具、什么时候先取时间、何时建定时任务、回答里不要暴露 SQL 等。

SQL 生成主提示词
由 build_sql_prompt() 动态拼出来，把 schema、同义词、敏感列、SQL 样本一起塞给模型。prompt_builder.py (line 74)
生效时机：

创建定时任务自动生成 SQL 时 scheduler_tool.py (line 159)
更新定时任务、只给 description 时 scheduler_manage_tool.py (line 145)
旧的 /api/v1/sql/generate 接口里也会走 sql_agent.py (line 31)
效果：影响 SQL 的字段选择、别名风格、是否参考样本、是否避开敏感列。
SQL 专用 system prompt
内容很短，核心就是“你是数据库查询助手，只返回 SQL”。sql_tool.py (line 174) scheduler_tool.py (line 163) scheduler_manage_tool.py (line 149)
生效时机：每次内部调 SQL LLM。

达到最大工具次数后的兜底总结提示词

四、我给你的改良建议

先做一次“配置瘦身”
把配置分成三类：

正在生效
预留但未接线
历史遗留
然后把未生效项要么接上，要么删掉。现在最大的问题不是少，而是“看起来能配，实际上没用”。
把后端 env 收口成一个统一 Settings
现在读取 env 比较散，os.getenv() 到处都是。
更稳的做法是：一个 AppSettings 统一接管，启动时打印“已生效配置摘要（脱敏）”。
这样一眼就知道：

读了哪个文件
最终值来自哪里
哪些变量压根没被消费
前端配置链统一一种口径
我建议二选一：

要么全走运行时 config.js
要么全走 Vite VITE_*
现在 APP_*、VITE_APP_*、window.__APP_CONFIG__ 三套同时存在，认知成本偏高，而且本地开发容易误判。
把 SQL 提示词集中成一个唯一真源
现在 SQL 相关 prompt 有重复和分叉：

build_sql_prompt() 一套
sql_tool.py 末尾又追加一套更强规则
schedule_task / manage_scheduled_task 并不完全一致
结果就是：聊天查数、建任务、改任务，三条链路生成 SQL 的风格和约束不完全一样。
这个很容易埋隐患。
要么真正启用 query_patterns，要么删掉
它本来是很好的“高频查询直达模板”机制，但现在基本没接到主流程。
如果接上，能降低延迟、提升稳定性；如果不接，继续放着只会误导维护者。

display_fields / required_fields 要么接到 SQL 生成，要么去掉
这块其实很像“面向业务展示的字段规范”，思路是对的。
但如果当前没参与 SQL 选择或回答格式控制，它就是静态摆设。

CHAT_API_TOKEN 这个点要尽快定性
现在文档里像是“可做鉴权”，但代码里没真正执行鉴权。
这会给人一种“好像有保护”的错觉。这个比没有还危险一点。

一句话总结：
现在最大的不是“配置少”或“提示词少”，而是“生效链路不够统一，死配置和半生效配置有点多”。
我建议先把“哪些真的生效”收口成一张清单，再做小步整理，效果会很明显。
