# 仓库协作规则

以下规则对本仓库内的代码修改默认生效。

## 提交规则

- 只要本次任务涉及代码修改，结束时必须创建一次 git 提交。
- 只提交本次任务相关文件，不把无关改动混入当前提交。
- 提交完成后，必须用自然语言说明本次修改内容、验证结果和提交信息。

## 回撤规则

- 用户说“回撤”“回退”“撤销最近版本”时，默认处理最近一次由当前执行者创建的提交。
- 若工作区干净且用户明确要求直接回到上一个版本，可直接回到最近一次提交前的状态。
- 若工作区不干净，或需要保留历史，优先使用保留历史的回撤方式，并说明原因。

## 注释规则

- 新增方法、函数或复杂逻辑块，需要补充适量注释。
- 注释应说明用途、关键约束、输入输出或为什么这样做。
- 已经足够直观的简单实现，不写冗余注释。
- 修改已有文件时，保留并维护原有合理的文件头说明注释和方法注释；若本次改动影响注释准确性，需要同步更新，不要直接删掉。

## 配置规则

- 若修改 .env.sample，同步检查并修改 .env；以 .env 中实际生效配置为准，必要时可用 .env 覆盖 .env.sample。

## 其他

- 不要回退他人未明确要求撤销的改动。
- 变更范围保持最小，优先简单可维护的实现。
- 不要改动明显不相关且原本正确的内容，例如已有文案、图标、提示语、超时提示等；若判断这些内容也需要优化，必须先征求用户确认。
