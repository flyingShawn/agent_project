### 服务关闭方法

#### 方法1：根据端口号关闭

**关闭后端服务（端口8000）**：

```bash
# 1. 查找进程ID
netstat -ano | findstr :8000
# 输出示例：TCP 0.0.0.0:8000 ... LISTENING 12345

# 2. 关闭进程（替换12345为实际进程ID）
taskkill /PID 12345 /F
```

**关闭Ollama服务（端口11434）**：

```bash
# 1. 查找进程ID
netstat -ano | findstr :11434

# 2. 关闭进程
taskkill /PID <进程ID> /F
```

**关闭Qdrant服务（端口6333）**：

```bash
# 1. 查找进程ID
netstat -ano | findstr :6333

# 2. 关闭进程
taskkill /PID <进程ID> /F
```

#### 方法2：根据进程名称关闭

```bash
# 关闭所有Python进程（谨慎使用！）
taskkill /IM python.exe /F

# 关闭所有Node进程（谨慎使用！）
taskkill /IM node.exe /F

# 关闭Ollama进程
taskkill /IM ollama.exe /F
```

### 端口冲突处理

#### 情况1：8000端口被占用

**检查是什么占用了端口**：

```bash
netstat -ano | findstr :8000
```

**情况A：之前启动的后端服务还在运行**

```bash
# 解决方案1：直接使用现有服务
# 访问 http://localhost:8000/api/v1/health 验证服务正常
# 如果正常，可以跳过启动步骤

# 解决方案2：关闭现有服务，重新启动
taskkill /PID <进程ID> /F
python -m uvicorn agent_backend.main:app --reload --host 0.0.0.0 --port 8000
```

**情况B：其他程序占用了8000端口**

```bash
# 解决方案1：关闭占用端口的程序
taskkill /PID <进程ID> /F

# 解决方案2：使用其他端口启动后端
python -m uvicorn agent_backend.main:app --reload --host 0.0.0.0 --port 8001
# 注意：使用其他端口后，前端配置也需要相应修改
```

#### 情况2：11434端口被占用（Ollama）

**检查是什么占用了端口**：

```bash
netstat -ano | findstr :11434
```

**情况A：Ollama服务已经在运行**

```bash
# 解决方案：直接使用现有服务
ollama list  # 验证服务正常
```

**情况B：其他程序占用了11434端口**

```bash
# 解决方案：关闭占用端口的程序
taskkill /PID <进程ID> /F

# 然后启动Ollama
ollama serve
```

#### 情况3：6333端口被占用（Qdrant）

**检查是什么占用了端口**：

```bash
netstat -ano | findstr :6333
```

**情况A：Qdrant服务已经在运行**

```bash
# 解决方案：直接使用现有服务
# 访问 http://localhost:6333/dashboard 验证服务正常
```

**情况B：其他程序占用了6333端口**

```bash
# 解决方案：关闭占用端口的程序
taskkill /PID <进程ID> /F

# 然后启动Qdrant
docker run -d -p 6333:6333 qdrant/qdrant
```

## Docker部署测试

### 前提条件

✅ 已安装 Docker Desktop
✅ Docker Desktop 正在运行

### 快速部署（一键启动）

#### Windows系统：

```bash
# 进入项目目录
cd d:\work_space\agent_project

# 运行部署脚本
docker\deploy.bat
```

#### Linux/Mac系统：

```bash
# 进入项目目录
cd /path/to/agent_project

# 添加执行权限
chmod +x docker/deploy.sh

# 运行部署脚本
./docker/deploy.sh
```

### 手动部署步骤

#### 第一步：检查配置

确保 `.env` 文件存在并已配置数据库连接。

#### 第二步：构建镜像

```bash
docker-compose build
```

#### 第三步：启动服务

```bash
docker-compose up -d
```

#### 第四步：查看服务状态

```bash
docker-compose ps
```

应该看到4个服务都在运行：

- `desk-agent-backend` (后端API)
- `desk-agent-frontend` (前端界面)
- `desk-agent-qdrant` (向量数据库)
- `desk-agent-ollama` (大模型服务)

#### 第五步：下载模型

**首次启动需要下载模型**（需要较长时间）：

```bash
# 进入Ollama容器
docker exec -it desk-agent-ollama bash

# 下载模型
ollama pull qwen2.5:7b
ollama pull qwen2.5-vl:7b

# 退出容器
exit
```

### 访问服务

- **前端界面**：http://localhost
- **后端API文档**：http://localhost:8000/docs
- **Qdrant控制台**：http://localhost:6333/dashboard

### 常用Docker命令

```bash
# 查看日志
docker-compose logs -f

# 查看特定服务日志
docker-compose logs -f backend

# 停止服务
docker-compose down

# 重启服务
docker-compose restart

# 清理所有数据（谨慎使用）
docker-compose down -v
```

## 常见问题排查

### 问题1：后端启动失败

**症状**：`ModuleNotFoundError: No module named 'xxx'`

**解决方案**：

```bash
# 重新安装依赖
pip install -r requirements.txt
```

### 问题2：Ollama连接失败

**症状**：`Ollama 流式调用失败`

**解决方案**：

1. 检查Ollama是否运行：访问 http://localhost:11434
2. 检查模型是否下载：`ollama list`
3. 检查 `.env` 中的模型配置：
   - ✅ 正确：`CHAT_MODEL=qwen2.5:7b`
   - ❌ 错误：`CHAT_MODEL=qwen2.5:7b-instruct`

### 问题3：Qdrant连接失败

**症状**：`Connection refused to localhost:6333`

**解决方案**：

```bash
# 使用Docker启动Qdrant
docker run -d -p 6333:6333 qdrant/qdrant
```

### 问题4：前端无法连接后端

**症状**：前端界面打开但无法发送消息

**解决方案**：

1. 检查后端是否运行：访问 http://localhost:8000/api/v1/health
2. 检查前端配置：`agent_frontend/vite.config.js` 中的代理设置
3. 检查浏览器控制台错误信息

### 问题5：Docker构建失败

**症状**：`docker-compose build` 失败

**解决方案**：

1. 检查Docker Desktop是否运行
2. 清理Docker缓存：`docker system prune -a`
3. 重新构建：`docker-compose build --no-cache`

### 问题6：模型下载慢

**症状**：`ollama pull` 很慢或失败

**解决方案**：

1. 使用国内镜像源（需要配置）
2. 使用代理
3. 使用更小的模型：`ollama pull qwen2.5:3b`

### 问题7：数据库连接失败

**症状**：`数据库查询失败`

**解决方案**：

1. 运行配置检查：`检查配置.bat`
2. 检查数据库服务是否启动
3. 检查 `.env` 中的数据库配置：
   ```bash
   DB_HOST=localhost
   DB_PORT=3306
   DB_NAME=your_database
   DB_USER=root
   DB_PASSWORD=your_password
   ```
4. 运行数据库连接测试：`python scripts/测试数据库连接.py`
